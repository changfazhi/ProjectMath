STRIPE INTEGRATION PLAN — ProjectMath
======================================
Branch: stripe-API-and-payment
Date: 2026-07-03

OVERVIEW
--------
Freemium model: monthly ($15/mo) and annual ($144/yr = $12/mo) subscriptions.
Premium gates: AI hints chatbot + photo-based AI grader + phone QR upload.
Uses Stripe Checkout (hosted payment page — no Stripe.js needed on frontend).
Tier propagated via Firebase custom claims (existing mechanism).

----------------------------------------------

- Price B: $144.00 / year recurring
- Save both Price IDs → go into .env as STRIPE_PRICE_MONTHLY and STRIPE_PRICE_ANNUAL


STEP 2 — Environment Variables
--------------------------------
Add to backend/.env (fill in manually):

  STRIPE_SECRET_KEY=sk_test_...          # Stripe Dashboard → Developers → API keys
  STRIPE_WEBHOOK_SECRET=whsec_...        # From running: stripe listen --forward-to localhost:3001/api/billing/webhook
  STRIPE_PRICE_MONTHLY=price_...         # From Step 1
  STRIPE_PRICE_ANNUAL=price_...          # From Step 1

Dev vs Production:
- Local .env uses sk_test_... + whsec_... from `stripe listen`
- Production env (hosting platform dashboard) uses sk_live_... + whsec_... from Stripe Dashboard webhook endpoint
- Never commit .env; never put live keys in .env

Running stripe listen (do this AFTER the backend is built):
  stripe listen --forward-to localhost:3001/api/billing/webhook
  → Prints: "Your webhook signing secret is whsec_abc123..."
  → Copy that into STRIPE_WEBHOOK_SECRET in .env
  → Keep the terminal open while testing

STEP 6 — Billing Service
--------------------------
New file: backend/src/services/billingService.ts

Functions:
  createCheckoutSession(userId, email, priceId)
    - Finds or creates Stripe Customer (reuse stripe_customer_id from users table)
    - Creates Stripe Checkout Session (mode: 'subscription')
    - Success URL: <origin>/roadmap?checkout=success
    - Cancel URL:  <origin>/roadmap?checkout=canceled
    - Returns { url }

  createPortalSession(userId)
    - Looks up stripe_customer_id for the user
    - Creates Stripe Customer Portal session
    - Returns { url }

  handleWebhookEvent(rawBody, signature)
    - Verifies Stripe signature using STRIPE_WEBHOOK_SECRET
    - Handles events:
        checkout.session.completed
          → setCustomUserClaims(firebaseUid, { tier: 'paid' })
          → UPDATE users SET subscription_status='active' WHERE id=userId

        customer.subscription.updated  (status = 'canceled' | 'past_due')
          → setCustomUserClaims(firebaseUid, { tier: 'free' })
          → UPDATE users SET subscription_status=<status>

        customer.subscription.deleted
          → setCustomUserClaims(firebaseUid, { tier: 'free' })
          → UPDATE users SET subscription_status='canceled'


STEP 7 — Billing Routes
-------------------------
New file: backend/src/routes/billing.ts

  POST /api/billing/checkout   requireAuth   body: { priceId }  → { url }
  POST /api/billing/portal     requireAuth                       → { url }
  POST /api/billing/webhook    NO auth, raw body                 → 200 OK

IMPORTANT: The webhook route uses express.raw({ type: 'application/json' })
to receive the unprocessed body for Stripe signature verification.
It must be mounted in index.ts BEFORE express.json() applies to it.

KNOWN CONSTRAINTS
-----------------
- Webhook route needs raw body — mounted before express.json() in index.ts
- Live keys must NEVER go in .env file; set them only in the hosting platform's env dashboard
- After gating features, free users lose AI access immediately — coordinate timing with launch
- Firebase custom claims are cached in the ID token (1-hour TTL) — force refresh on success redirect resolves this instantly for the paying user; other sessions auto-refresh within 1 hour
