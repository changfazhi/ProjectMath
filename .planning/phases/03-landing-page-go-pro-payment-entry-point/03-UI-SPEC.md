> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

**Scope note:** This phase introduces **no new visual surface**. It rewires which JS handler fires when the landing page's "Go Pro" CTA is clicked, and reuses the existing `LoginModal` and `UpgradeModal` components verbatim (per `PROJECT.md` Key Decisions: "Reuse the existing globally-mounted `UpgradeModal`... avoids duplicating checkout logic"). ## Interaction Contract (Phase-Specific)

This is the binding contract for this phase's implementation.

### 1. "Go Pro" CTA ‚Äî preserve visual treatment, change only the handler

- The CTA stays the exact same `<a>` element at `LandingPage.tsx:402`, with its existing `class="pm-hov-lift"` and existing inline `style` (gradient background, `13px` padding, `12px` border-radius, `700` weight, `15px` size, `transform .15s` hover-lift) **untouched**.
- It currently has `href="#"` and is unhandled by `handleClick()` (falls through to browser-default `#` scroll-to-top ‚Äî the bug this phase fixes).
- Follow the established delegation pattern already used for the nav "Log in" link (`data-login` attribute, `LandingPage.tsx:499-501`, `:518`): add a new data attribute (e.g. `data-goto-pro`) to the CTA and branch on it in `handleClick()`, calling `e.preventDefault()` so the dead `#` scroll no longer fires.
- Do not swap the `<a>` for a `<button>` and do not add/remove any class or inline style as part of this rewiring ‚Äî the hover-lift and gradient must look and behave identically to today, only the click outcome changes.

### 2. Logged-in visitor clicks "Go Pro"

- `UpgradeModal` opens immediately via the existing `openUpgradeModal()` (`AuthContext.tsx:75`) ‚Äî no navigation, no intermediate screen, no loading affordance needed (this mirrors the Header "Get Premium" button's existing direct-open behavior ‚Äî `Header.tsx:78`).

### 3. Logged-out visitor clicks "Go Pro" ‚Äî login-then-upgrade sequencing

Order of operations (all reusing existing components/APIs, no new UI):

1. Click sets a **pending intent** (e.g. a ref or state flag scoped to `LandingPage.tsx`, mirroring the existing `prevUserRef`/`justLoggedIn` pattern at `LandingPage.tsx:484-489`) marking "this login is for Go Pro."
2. `openLoginModal('Log in to continue to Premium checkout.')` opens `LoginModal` with the tailored message (see Copywriting Contract). Visually and behaviorally `LoginModal` is 100% unchanged ‚Äî same tabs, same Google/email flows, same loading states on its own buttons ("Signing in‚Ä¶" / "‚Ä¶").
3. On successful sign-in, `LoginModal` closes itself exactly as today (`setShowLogin(false)` inside `signInWithGoogle`/`signInWithEmail`/`signUp` in `AuthContext.tsx`) ‚Äî **no change** to that close mechanism.
4. When the `user` transition effect in `LandingPage.tsx` (`:484-489`) fires `justLoggedIn`:
   - If the pending intent is "Go Pro" ‚Üí call `openUpgradeModal()` and **do not** `navigate('/roadmap')`. The visitor stays on the landing page with `UpgradeModal` now open over it.
   - Otherwise (plain nav "Log in") ‚Üí **unchanged**: `navigate('/roadmap', { replace: true })`, exactly as today. This is the regression guard for Success Criterion 4.
5. Clear the pending intent flag once consumed (whichever branch fires) so a later plain sign-out/sign-in cycle doesn't incorrectly re-trigger the upgrade modal.

### 4. Transition affordance between the two modals

- No new spinner, skeleton, or transitional screen is introduced. The brief interval between `LoginModal` closing and `UpgradeModal` opening (bounded by the existing `onAuthStateChanged` listener firing, typically sub-second) is acceptable to leave as a plain view of the landing page ‚Äî this matches how the Header's direct "Get Premium ‚Üí `UpgradeModal`" open already behaves with no loading affordance today.
- Do **not** add a global loading overlay, toast, or redirect flash for this transition ‚Äî that would be new visual surface, out of scope for this phase.
- If in implementation the gap is empirically jarring (e.g. modal-less flicker longer than ~300ms), the acceptable fallback ‚Äî still with no new component ‚Äî is to keep `LoginModal` mounted with its existing internal button-level "Signing in‚Ä¶" loading text until the `user` transition effect fires, then swap directly to `UpgradeModal`. This must not introduce a new dedicated "loading" component.

### 5. No new components needed

- Confirmed: this phase requires zero new components. It only threads a pending-intent flag through `LandingPage.tsx` (and, if the planner decides, a small optional parameter on `openLoginModal`/the auth context ‚Äî an implementation detail, not a new visual component) and adds one new delegated click branch in `handleClick()`.
- `LoginModal.tsx` and `UpgradeModal.tsx` source and props are **not** to be modified for styling purposes in this phase (functional prop additions like an optional post-login callback are in scope if the planner's chosen implementation needs them; visual changes are not).
