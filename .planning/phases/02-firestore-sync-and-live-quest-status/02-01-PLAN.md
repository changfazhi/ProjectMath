<objective>
Make a signed-in student's daily study plan persist in Firestore (keyed to their Firebase UID, one date-keyed document per day) so it survives a page reload and appears on any other device, while anonymous users keep the existing localStorage behaviour with zero regression. localStorage becomes a fast-load cache layered under a Firestore-first read.

Purpose: Phase 2 success criteria 1 and 2 — cross-device persistence for authenticated users plus a graceful localStorage fallback for everyone else.
Output: A Firestore client, Firestore-aware persistence helpers in the shared studyPlan lib, and read/write wiring in the sidebar hook and the plan-generation page.
</objective>

## Phase Goal

**As a** signed-in H2 Math student, **I want to** have my generated daily study plan saved to the cloud under my account, **so that** I see the same plan in the sidebar on any device I log in from — and if I'm not signed in it still works locally.

    In `studyPlan.ts`, keep the existing `StoredPlan`, `PLAN_KEY`, `todayStr`, `savePlan`, `loadStoredPlan` exports unchanged (they remain the localStorage cache layer). Add four new async helpers using the modular Firestore SDK (`doc`, `getDoc`, `setDoc` from `firebase/firestore`) and the `db` export:

    - `savePlanRemote(uid: string, plan: StoredPlan): Promise<void>` — writes `plan` to document path `users/{uid}/study_plans/{plan.date}` via `setDoc`. The collection is `study_plans`, the document id is the `YYYY-MM-DD` date string (so there is exactly one plan per user per day, matching the one-plan-per-day requirement). Store the full `StoredPlan` shape (date, items, reasoning) so a remote read reconstructs an identical object.
    - `loadRemotePlan(uid: string): Promise<StoredPlan | null>` — reads `users/{uid}/study_plans/{todayStr()}` via `getDoc`; returns the typed `StoredPlan` when the doc exists and its `items` is an array, else `null`. Validate the same way `loadStoredPlan` validates (Array.isArray(items)).
    - `persistPlan(uid: string | null, plan: StoredPlan): Promise<void>` — ALWAYS calls `savePlan(plan)` (localStorage cache, fast subsequent loads). When `uid` is non-null, ALSO awaits `savePlanRemote(uid, plan)`. This satisfies PERS-03's "write to both" cache strategy and PERS-02's anonymous-stays-local rule (no remote write without a uid).
    - `resolvePlan(uid: string | null): Promise<{ plan: StoredPlan | null; isStale: boolean }>` — Firestore-first when signed in: if `uid` is non-null, try `loadRemotePlan(uid)`; on a hit, write it through to localStorage via `savePlan` (refresh the fast-load cache) and return `{ plan, isStale: false }` (a remote doc is always today's by construction since it is keyed by `todayStr()`). If the remote read returns null OR throws (offline / rules / transient), fall back to `loadStoredPlan()`. When `uid` is null, return `loadStoredPlan()` directly (PERS-02 anonymous path, unchanged behaviour). Wrap the remote read in try/catch so a Firestore failure degrades to the cached/local plan rather than surfacing an error.

    No backslash/JSON escaping concerns here — this is TypeScript, not a SQL migration. Strict TS: no `any`; type the Firestore doc data through `StoredPlan`.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser client -> Firestore | The Firebase client SDK writes/reads study-plan docs directly from the browser; there is no Express proxy. Access control lives entirely in Firestore security rules. |
| useAuth() UID -> doc path | The signed-in user's UID selects which `users/{uid}/study_plans/{date}` document is read/written. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-02-01-I | Information Disclosure | `loadRemotePlan` reading `users/{uid}/study_plans/*` | high | mitigate | Firestore security rules MUST restrict reads of `users/{uid}/**` to `request.auth.uid == uid` (user_setup). Client never reads another UID's path because the path is built from `useAuth().user.uid`. |
| T-02-01-T | Tampering | `savePlanRemote` writing another user's plan doc | high | mitigate | Same ownership rule (`request.auth.uid == uid`) on writes; `persistPlan` passes only the current session UID. |
| T-02-01-D | Denial of Service | Firestore unavailable / offline | low | mitigate | `resolvePlan` wraps the remote read in try/catch and falls back to the localStorage cache, so the sidebar still renders. |
| T-02-01-S | Spoofing | Anonymous user writing to Firestore | medium | mitigate | `persistPlan` writes remotely only when `uid` is non-null; security rules deny unauthenticated writes. |
</threat_model>
